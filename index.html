Dual Role in Vera Rubin

🖥️

CPU Compute Rack

256 Vera CPUs in a liquid-cooled MGX rack. Purpose-built for reinforcement learning environments — simulating, testing, and validating agentic AI outputs at scale. Integrated with Spectrum-X for tight synchronisation across the AI factory.

💾

Inside BlueField-4

Vera CPU is embedded in the BlueField-4 DPU alongside ConnectX-9. This gives the STX storage rack its own ARM compute — enabling dedicated KV cache processing without burdening GPU nodes.

🖥️

NVL72 Host CPU

36 Vera CPUs sit alongside 72 Rubin GPUs in the NVL72 rack, providing host processing for the GPU compute nodes. Unified ARM architecture across the entire POD.

🎯 CTO Takeaway: ARM-First AI Factory

Vera makes the AI factory entirely ARM-based — no x86 anywhere in the POD. This eliminates x86 licensing costs, reduces power, and simplifies software stack management. If your existing infrastructure is x86-heavy, plan for a parallel ARM tooling investment (compilers, profiling, monitoring agents).

📐 Architecture Detail

NVLink 6 connects 72 Rubin GPUs + 36 Vera CPUs in the NVL72 rack via the NVLink 6 Switch. This creates a single shared memory domain where any GPU can directly access any other GPU's memory — critical for large MoE models where expert parameters must be rapidly shuffled across GPUs.

🎯 CTO Takeaway: Scale-Up vs Scale-Out

NVLink 6 is your scale-up fabric (within rack). Spectrum-6 / Quantum-X800 is your scale-out fabric (rack-to-rack). The NVL72 domain is the "sweet spot" — large enough to hold most models in-memory, small enough for full NVLink bandwidth. When planning capacity, think in multiples of NVL72 (72-GPU units), not individual servers.

ConnectX-9 SuperNIC

Ultra-low-latency network interface controller — the GPU's direct connection to the scale-out network.

🌐

Scale-Out NIC

ConnectX-9 provides each Rubin GPU with a dedicated high-bandwidth path to the network fabric. This eliminates the bottleneck of sharing NICs across multiple GPUs, ensuring linear scaling for distributed training.

🔗

Inside BlueField-4

ConnectX-9 is also embedded within the BlueField-4 DPU, giving the STX storage rack its own high-bandwidth network path for KV cache retrieval and storage operations.

⚡

Offload Engine

Hardware offloads for collective operations (all-reduce, all-gather) reduce CPU overhead and improve network utilisation during distributed training runs.

🎯 CTO Takeaway: NIC-per-GPU is Table Stakes

ConnectX-9 continues Nvidia's "one NIC per GPU" design philosophy. When comparing to AMD's MI400 or Intel's Gaudi3, verify they offer equivalent NIC density — shared NIC architectures become bottlenecks at >10k GPU scale.

🎯 CTO Takeaway: Go Ethernet Unless You're Building Stargate

Spectrum-X Ethernet is now "good enough" for most AI workloads, including large-scale training. InfiniBand's latency advantage matters only at extreme scale (>10k GPUs) or for HPC workloads. The management simplicity and multi-vendor compatibility of Ethernet reduces TCO significantly. Choose InfiniBand only if you're building a national-scale AI factory or have strict HPC requirements.

BlueField-4 & STX Storage

AI-native storage infrastructure that extends GPU memory across the POD — purpose-built for the KV cache problem in agentic AI.

💾

BlueField-4 DPU

Combines the Vera CPU + ConnectX-9 SuperNIC on a single card. The DPU offloads storage, networking, and security tasks from the host CPU — enabling "zero-touch" infrastructure processing.

🗂️

STX Storage Rack

BlueField-4 STX is a rack-scale context memory system. It extends GPU memory seamlessly across the entire POD, providing a high-bandwidth shared layer for KV cache data generated by LLMs and agentic workflows.

📝

DOCA Memos

A new DOCA framework that supercharges BlueField-4 storage. Enables dedicated KV cache storage processing — boosting inference throughput by up to 5× while significantly improving power efficiency vs. general-purpose storage.

🎯 CTO Takeaway: The KV Cache Problem is Real

As context windows grow (1M+ tokens in agentic AI), KV cache becomes the dominant memory consumer. Traditional approaches — offloading to CPU RAM or SSD — add latency and complexity. STX creates a dedicated tier between GPU HBM and general storage, purpose-built for KV cache. If you're serving long-context models, this alone justifies the Vera Rubin platform over alternatives.

How LPX and Rubin Work Together

🔑 Joint Decode: GPUs + LPUs Compute Every Token Together

When deployed with Vera Rubin NVL72, Rubin GPUs and Groq LPUs jointly compute every layer of the AI model for every output token. This is not a separate inference cluster — it's a codesigned system where GPUs handle the parts they're best at (attention, MoE routing) and LPUs handle deterministic, low-latency decode. The result is far higher throughput per watt than either processor alone.

🎯 CTO Takeaway: Inference-Optimised SKU

Groq 3 LPX rack is for ultra-premium inference — trillion-parameter models with million-token context. If your business is serving GPT-4-class or larger models at scale, the 35× throughput/MW improvement makes LPX a compelling addition. For smaller models or training-only workloads, the NVL72 rack alone is sufficient.

DSX & AI Factory Design

Nvidia's datacenter infrastructure platform — integrating power, cooling, and system management for Vera Rubin at AI factory scale.

🔋

DSX Max-Q

Dynamic power provisioning across the entire AI factory. Enables deploying 30% more AI infrastructure within a fixed-power datacenter by dynamically shifting power between compute and cooling based on workload.

⚡

DSX Flex

Makes AI factories grid-flexible assets — scaling power consumption up/down based on grid conditions. Nvidia claims this unlocks 100 GW of stranded grid power worldwide. Critical for sites with variable renewable energy.

📐

AI Factory Reference Design

A blueprint for codesigned AI infrastructure that maximises tokens per watt and overall goodput. Tightly integrates compute, networking, storage, power, and cooling — ensuring AI factories can scale reliably under continuous high-intensity workloads.

🎯 CTO Takeaway: Power is the Real Bottleneck

GPU availability is no longer the constraint — power and cooling are. DSX Max-Q's 30% density improvement within fixed power is a big deal: it means you can fit 30% more compute into your existing power envelope without building new datacenters. DSX Flex's grid-flexible model is strategic for sites with renewable energy — run hard when wind/solar is abundant, throttle when it's not.

200+ Infrastructure Partners

Nvidia is building DSX with 200+ datacenter infrastructure partners. This is not a proprietary woodoo — it's an industry reference architecture. Key partners include power distribution, cooling, cabling, and rack vendors who build to the DSX specification.

🎯 CTO Takeaway: Platform Lock-In vs Multi-Vendor

Vera Rubin's biggest strength — deep codesign across 7 chips — is also its biggest risk: total Nvidia lock-in. AMD's MI400 + UALink + Ultra Ethernet Consortium approach offers multi-vendor interoperability but lacks the integration depth. Your decision comes down to: do you prioritise maximum tokens per watt (Vera Rubin) or supply chain resilience and multi-sourcing (AMD + UALink)? For most AI cloud companies, the answer is both — run Vera Rubin for latency-critical inference, AMD for training capacity diversification.

🎯 CTO Takeaway: Dynamo is the Dark Horse

NVIDIA Dynamo is arguably the most strategically important piece of the Vera Rubin software stack. It's the orchestration layer that makes GPU+LPX+STX work as a single system — routing prefill to GPUs, decode to LPX, and KV cache to STX. Without Dynamo, you just have three separate hardware systems. With it, you have a codesigned inference supercomputer. Prioritise Dynamo evaluation and team training.
