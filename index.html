Enterprise RA Configuration

📐 2-8-9-800 Node Architecture

The Enterprise RA uses a 2-8-9-800 nomenclature: 2 CPUs, 8 GPUs, 9 NICs (8× ConnectX-8 + 1× BlueField-3), 800 Gb/s per GPU. Systems are organised into Scalable Units (SUs) of 4 partner servers each, providing rapid deployment of multiple cluster sizes.

🖥️

HGX B300 Server

8× B300 GPUs with NVLink 5 full-mesh interconnect. 2× Grace or Intel GNR CPUs. 8× ConnectX-8 SuperNICs for scale-out. 1× BlueField-3 DPU for management and offload. The standard AI training server.

🔗

NVLink 5 Switch Tray

Enables the 72-GPU NVL domain within the NVL72 rack. Full-bandwidth GPU-to-GPU communication. Critical for large-model training where parameters must be shuffled across GPUs rapidly.

🌐

Spectrum-X Networking

Spectrum-4 Ethernet platform with SN5600/SN5610 switches. Multi-plane and converged topologies for 1K–16K GPU clusters. BlueField-3 DPU acceleration for in-network compute.

🎯 CTO Takeaway: Deploy Now, Plan for VR

GB300 NVL72 is the platform you can deploy today. If you need capacity now, don't wait for Vera Rubin — but design your infrastructure (power, cooling, network) to be VR-ready. Key differences: VR uses ConnectX-9 and BlueField-4, but the rack footprint and DLC requirements are similar. Plan for a rolling upgrade path from B300 to VR NVL72.

📐 Architecture Detail

NVLink 6 connects 72 Rubin GPUs + 36 Vera CPUs in the NVL72 rack via the NVLink 6 Switch. This creates a single shared memory domain where any GPU can directly access any other GPU's memory — critical for large MoE models where expert parameters must be rapidly shuffled across GPUs.

🎯 CTO Takeaway: Scale-Up vs Scale-Out

NVLink 6 is your scale-up fabric (within rack). Spectrum-6 / Quantum-X800 is your scale-out fabric (rack-to-rack). The NVL72 domain is the "sweet spot" — large enough to hold most models in-memory, small enough for full NVLink bandwidth. When planning capacity, think in multiples of NVL72 (72-GPU units), not individual servers.

Converged vs Multi-Plane

🔀

Converged Network

Compute, HPS storage, and in-band management share the same Ethernet fabric. Lower switch count (~40% fewer), simpler cabling, faster deployment. Use BlueField-3 DPU acceleration for traffic isolation. Ideal for NCP multi-tenant environments.

🔗

Multi-Plane (Dedicated)

Separate physical networks for compute, HPS, and management. Maximum performance and isolation. Required for >8K GPU deployments with demanding HPS requirements. More switches, more cabling, but no contention between traffic classes.

Network Resiliency

🛡️

EVPN-MH (Multihoming)

Ethernet VPN Multihoming provides redundant host connectivity. If one leaf switch fails, traffic automatically reroutes via the alternate leaf. Standard for converged network deployments.

🔀

ECMP with BlueField DPU

Hardware-accelerated Equal Cost Multi-Path routing. BlueField DPU mode enables line-rate ECMP without CPU overhead. Critical for large-scale training where all-reduce traffic must be load-balanced across multiple paths.

🔗

MLAG (Multi-Chassis LAG)

Multi-Chassis Link Aggregation for redundant uplinks. Provides active-active connectivity to spine switches. Alternative to EVPN-MH for environments that prefer L2-based resiliency.

🎯 CTO Takeaway: Start Converged, Go Multi-Plane at Scale

For <4K GPUs, converged networking is simpler and cheaper. Beyond 8K GPUs, multi-plane becomes necessary to avoid contention. BlueField-3 DPU acceleration is the key enabler for converged networks — it provides hardware isolation between traffic classes without separate physical fabrics. Budget for BlueField-3 in DPU mode, not NIC mode.

📊 Key Benchmarking Results

  • Latency drops as systems scale — more GPUs = faster per-query response
  • Reasoning model scale impacts latency — larger models need more compute per step
  • Use NeMo Agent Toolkit for benchmarking and sizing your specific workload

🎯 CTO Takeaway: Budget for the Reasoning Overhead

AI-Q's agentic approach means significantly more compute per query than simple RAG. Each query may trigger multiple retrieval steps, reasoning iterations, and tool calls. Budget 2–4× the GPU capacity compared to a simple RAG pipeline. Use the NeMo Agent Toolkit to benchmark your specific workload before committing to hardware.
